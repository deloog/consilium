# 实验项目问题修复报告

## 📋 问题总结

### 问题1：流水线模式 - 技术负责人输出缺失
**症状**：[pipeline_result.txt:79-81](pipeline_result.txt#L79-L81) 技术负责人的输出完全为空

**根本原因**：
1. `max_tokens=2000` 不足 - 设计师输出较长(约1500 tokens)，技术负责人接收长上下文后没有足够空间生成完整回复
2. 缺少错误处理 - API调用失败或token限制时没有任何提示
3. 没有针对不同角色的差异化配置

### 问题2：社交模式 - 所有讨论内容丢失
**症状**：[social_result.txt](social_result.txt) 所有角色的发言内容都是空的，只有名称标签

**根本原因**：
1. **致命问题**：`max_tokens=500` 严重不足 - 无法生成任何有意义的内容
2. **上下文膨胀**：每轮讨论传递完整历史，第5轮时输入可能达3000+ tokens，但输出只有500 tokens
3. **无历史管理**：没有截断、摘要或滑动窗口机制
4. **总结失败**：最终总结阶段也因为上下文过长+token不足而失败

## 🎯 核心改进原理

### Token预算分配策略

**修复前的问题**：
```
社交模式第5轮：
输入上下文 = 需求(200) + 20条发言×平均150字 = 3200+ tokens
输出空间 = 500 tokens ❌ 模型被迫输出空内容或截断
```

**修复后的策略**：
```
社交模式第5轮：
输入上下文 = 需求(200) + 最近12条发言×平均150字 = 2000 tokens ✅
输出空间 = 1000 tokens ✅ 模型有充足空间生成完整回复
```

### 上下文管理策略

**滑动窗口**：
```
第1轮：需求 + 4条发言 (总5条)
第2轮：需求 + 8条发言 (总9条)
第3轮：需求 + [摘要] + 最近12条 (控制在15条以内)
第4轮：需求 + [摘要] + 最近12条 (控制在15条以内)
第5轮：需求 + [摘要] + 最近12条 (控制在15条以内)
```

## ⚠️ 注意事项

1. **API费用**：修复后每次调用消耗更多tokens，请注意API额度
2. **模型限制**：GLM-4.7的最大上下文长度有限，如果讨论非常复杂，可能仍需调整
