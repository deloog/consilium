# 多智能体协作模式对比实验

## 实验目标

公平对比两种多智能体协作模式的真实能力上限：
- **流水线模式（Pipeline Mode）**：顺序传递，每个角色只能看到上一个角色的输出
- **社交讨论模式（Social Discussion Mode）**：自由讨论，所有角色可以互相质疑和补充

## 核心假设

**流水线模式的特点：**
- ✅ 每个角色可以深度专注于自己的领域
- ✅ 输出更专业、更完整
- ❌ 容易在错误的前提上建造"精致的大厦"
- ❌ 缺乏跨角色的纠错机制

**社交讨论模式的特点：**
- ✅ 能更早发现和修正方向性错误
- ✅ 跨角色的碰撞能产生新想法
- ❌ 可能过度保守（讨论容易走向"砍砍砍"）
- ❌ 上下文变长后容易出现重复和趋同

## 改进措施（对称性增强）

为了公平对比，对两种模式都进行了**对称性增强**：

### 流水线模式增强
1. **自我审视环节**：每个角色输出后，质疑自己的方案
2. **原始需求可见**：所有角色都能看到原始需求，避免信息漂移
3. **内部评审环节**：模拟真实的方案评审会

### 社交讨论模式增强
1. **智能压缩策略**：保留关键决定，避免重要信息丢失
2. **阶段性回顾**：每2轮讨论后进行回顾，避免讨论跑偏
3. **重复检测**：检测并警告重复发言（但不阻止，让真实问题暴露）
4. **内部评审环节**：与流水线模式相同的评审流程

### 统一评估标准
6个维度的客观评分（每项0-10分）：
1. 需求理解准确性
2. 方向性错误风险
3. 可执行性
4. 创新与保守的平衡
5. 细节完整性
6. 用户价值

## 实验设计原则

### 1. 公平性（Fairness）
- 两种模式使用相同的需求、相同的角色设定
- 两种模式都进行了对称性增强
- 使用统一的评估标准

### 2. 真实性（Authenticity）
- 不掩盖缺陷，让每种模式的问题充分暴露
- 重复发言只警告不阻止（社交模式）
- 自我审视环节不修正方案，只指出问题（流水线模式）

### 3. 客观性（Objectivity）
- 评估者不知道方案来自哪种模式（盲评）
- 使用低温度（0.3）进行评估，减少随机性
- 6个维度的量化评分

### 4. 可复现性（Reproducibility）
- 所有代码开源
- API调用次数和温度参数透明
- 随机种子可固定（如需要）

## 观察重点

### 流水线模式
观察点：
- [ ] 是否会在错误的前提上深入执行？
- [ ] 自我审视环节能否发现问题？
- [ ] 内部评审能否起到兜底作用？
- [ ] 方案的专业性和完整性如何？

### 社交讨论模式
观察点：
- [ ] 能否在早期发现方向性错误？
- [ ] 是否会过度保守（砍太多需求）？
- [ ] 重复发言问题是否严重？
- [ ] 阶段性回顾能否缓解讨论跑偏？

### 对比分析
关键问题：
- 哪种模式在各维度得分更高？
- 哪种模式的最致命问题更严重？
- 在真实项目中，你会选择哪种模式？

## API 消耗预估

### 流水线模式
- 方案输出：3次（产品经理、设计师、技术负责人）
- 自我审视：3次
- 内部评审：1次
- **总计：7次**

### 社交讨论模式
- 讨论发言：5轮 × 4人 = 20次
- 阶段回顾：2次（每2轮1次）
- 会议总结：1次
- 内部评审：1次
- **总计：24次**

### 统一评估
- 评估流水线方案：1次
- 评估社交方案：1次
- 对比分析：1次
- **总计：3次**

**完整实验总计：34次 API 调用**

## 实验结果分析建议

1. **先看评估报告**：打开 `evaluation_result.txt`，查看【最终对比分析】部分
2. **对比得分**：看两种模式在6个维度的得分差异
3. **阅读原文**：对得分差异大的维度，回到原始方案对比细节
4. **思考原因**：分析为什么某种模式在某个维度表现更好/更差
5. **记录观察**：记录你的发现，为下一轮实验提供方向
