# 混合模式 v3.0 设计方案

## 📊 v2.0 的问题分析

### 问题1：**"实时美颜"的范围蔓延**（最致命）

**评估官原话：**
> "技术方案中的'实时美颜'功能依赖人脸检测，这属于未经批准的AI能力，是明确的范围蔓延。"

**问题根源：**
- **阶段0决策**：确定"砍掉AI，只保留基础美化"
- **阶段1流水线**：技术负责人理解"基础美化"为"实时美颜+人脸检测"
- **交叉审视环节**：设计师审视技术方案时，**没有发现**这是范围蔓延
- **阶段2质检**：质检会议**也没有发现**

**为什么没发现？**
1. 交叉审视环节的检查清单不够明确
2. 阶段2质检会议的分工不够细化
3. 缺少"与决策结论对照"的环节

---

### 问题2：预算评估仍然过于乐观

**评估官原话：**
> "部分成本评估过于乐观（云服务、测试成本），存在执行风险。"

**问题根源：**
- 阶段0的成本核算专家只能评估"大方向"的成本
- 阶段1技术方案的细节成本（云流量、测试）没有被充分审查
- 阶段2质检会议中，刘总应该重点审查技术成本，但可能力度不够

---

### 问题3：质检环节的检查不够深入

**评估官暗示：**
> "需要在流水线末端设置质检环节，以确保执行产出不偏离决策决议。"

**现状分析：**
- v2.0 的阶段2质检会议只有1轮，4个人各自发言
- 没有形成"逐项对照检查"的机制
- 缺少"强制停止并修正"的权力

---

## 🎯 v3.0 核心改进策略

### 改进1：增加"决策对照检查"环节

**插入位置：** 阶段1流水线结束后，阶段2质检开始前

**执行方式：**
```python
DECISION_COMPLIANCE_CHECK = {
    "name": "决策对照检查官",
    "system_prompt": """你是决策对照检查官，负责确保流水线输出不偏离决策结论。

你的任务：
1. 逐项对照：将技术方案与阶段0的决策结论逐条对比
2. 发现偏差：标记所有偏离决策的部分（如"决定砍AI，但技术方案有人脸检测"）
3. 判断性质：区分"合理细化"和"范围蔓延"

输出格式：
- 符合决策：[列出符合的部分]
- 可疑偏差：[列出需要讨论的偏差]
- 明确违背：[列出必须修正的部分]

控制在200字内。"""
}
```

**预期效果：**
- 在质检会议前，先由专门角色进行逐项对照
- "实时美颜"会被标记为"明确违背"
- 然后进入质检会议强制讨论修正

---

### 改进2：阶段2质检会议增加"强制修正"权力

**当前问题：**
- v2.0 的质检会议只是"指出问题"
- 没有"修正并重新审视"的机制

**v3.0 方案：**
```python
# 如果质检会议发现致命问题，触发"修正流程"
if 发现范围蔓延 or 预算严重超支:
    # 修正环节
    1. 技术负责人立即修正技术方案
    2. 成本核算专家重新评估预算
    3. 再次进入质检会议审查
```

**实现方式：**
- 质检会议后，增加一个"判断是否需要修正"的逻辑
- 如果需要，让相关角色重新输出
- 最多允许1次修正

---

### 改进3：成本核算专家参与技术方案审视

**当前问题：**
- v2.0 的交叉审视只有设计师和技术负责人
- 成本核算专家只在阶段0和阶段2出现
- 导致技术方案的成本细节没有被及时审查

**v3.0 方案：**
```python
COST_REVIEW_ROLE = {
    "name": "成本核算专家·刘总",
    "task": "审视技术方案的成本细节",
    "system_prompt": """你是成本核算专家，现在审视技术方案的预算分配。

关注点：
- 云服务成本（存储、流量、CDN）是否被低估？
- 第三方服务费用（如API调用）是否有隐藏成本？
- 测试成本（多机型、多场景）是否被充分评估？
- 总预算50万，已分配的成本是否合理？

如果发现严重低估，明确指出。
控制在150字内。"""
}
```

**插入位置：** 阶段1流水线结束后，作为第3次交叉审视

---

### 改进4：阶段0讨论增加"决策清单"输出

**当前问题：**
- 阶段0的总结是自然语言描述
- 缺少"清晰的禁止清单"

**v3.0 方案：**
```python
phase0_summary_prompt = """请总结决策结论，并输出：

1. 确认保留的核心功能（列表）
2. 明确砍掉的功能（列表）
3. 明确禁止的技术（如"禁止使用人脸检测、AI接口"）
4. 预算和工期的基本共识

格式化输出，便于后续检查。"""
```

**预期效果：**
- 阶段0输出一个"决策清单"
- 后续的"决策对照检查官"可以逐条核对

---

## 🏗️ v3.0 完整流程

```
阶段0：需求决策会议（3轮社交，4人）
  参与者：产品、用户、技术、成本核算
  输出：决策清单（格式化）

    ↓

阶段1：流水线模式（3个角色顺序输出）
  产品经理 → 设计师 → 技术负责人
  输出：PRD、设计方案、技术方案

    ↓

【新增】决策对照检查（1个专门角色）
  检查官逐项对照决策清单
  标记"符合"、"可疑"、"违背"

    ↓

交叉审视环节（3次审视）
  1. 设计师审视技术方案
  2. 技术负责人审视设计方案
  3. 【新增】成本核算专家审视技术成本

    ↓

阶段2：质检会议（1轮社交，4人）
  重点讨论"决策对照检查"发现的问题

    ↓

【新增】判断是否需要修正
  如果有致命问题（范围蔓延、预算超支）
  → 技术负责人修正技术方案
  → 重新进入质检会议

    ↓

最终评审（内部评审官）
```

## 🎯 v3.0 的核心价值

### 价值1：从"发现问题"到"强制修正"

**v2.0：**
- 质检会议发现问题
- 最终评审官指出问题
- 但**没有修正**

**v3.0：**
- 发现问题后，**立即修正**
- 修正后**重新审视**
- 形成真正的"闭环"

---

### 价值2：三道防线防止范围蔓延

**第一道防线：决策清单**
- 明确"禁止清单"（如"禁止人脸检测"）

**第二道防线：决策对照检查**
- 流水线结束后，逐项对照
- 标记偏差

**第三道防线：质检会议**
- 重点讨论偏差
- 强制修正

---

### 价值3：成本把关贯穿始终

**阶段0：** 成本核算专家参与决策
**阶段1：** 成本核算专家审视技术成本
**阶段2：** 成本核算专家参与质检

**三次把关，避免预算超支**

---

## ⚠️ 实施风险

### 风险1：流程复杂度增加

**API调用次数：**
- v2.0: 23次
- v3.0: 约27-29次
  - 决策对照检查：1次
  - 成本审视：1次
  - 修正流程（如果触发）：2-4次

**缓解措施：**
- 只在发现致命问题时才触发修正
- 最多允许1次修正

---

### 风险2：修正可能导致讨论循环

**如果修正后仍有问题怎么办？**

**方案：**
- 最多允许1次修正
- 如果修正后仍不通过，直接判定"方案失败，需重新启动"
- 避免无限循环
